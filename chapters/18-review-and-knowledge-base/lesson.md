# 学习

本章不再追新功能，而是整理你已经经历过的 AI coding 反馈。

## 1. 创建复盘 Issue

复制下面的提示词给 AI：

```text
请通过 gh CLI 创建一个 GitHub Issue，用来跟踪“课程复盘与知识沉淀”。

这个 Issue 只做复盘和整理，不直接开发新功能。

请在 Issue 中记录：

- 需要回顾哪些章节；
- 需要整理哪些 Issue、规格、检查点、diff 和测试经验；
- 需要沉淀哪些检查清单；
- 需要沉淀哪些提示词；
- 需要总结哪些范围扩张判断规则；
- 不新增业务功能、不重写前面章节、不建设复杂文档系统、不替换现有工具、不做完整团队知识库治理、不深入 UI/UX、产品方法论或组织流程；
- 需要形成最终交付物和自检问题。

创建前请先给我看标题和正文，等我确认后再运行 gh 命令。
```

## 2. 回顾课程主线

让 AI 帮你按主线整理：

```text
请基于课程复盘与知识沉淀 Issue，先不要写代码。

请帮我按下面线索复盘：

1. 从需求到 Issue；
2. 从 Issue 到规格；
3. 从规格到验证检查点；
4. 从 AI coding 到运行反馈；
5. 从 diff 到影响判断；
6. 从测试失败到修正；
7. 从一次经验到检查清单。

请用非开发人员能理解的语言输出。
```

这一步是把散乱经历重新串成一条工程思维链路。

## 3. 沉淀检查清单

让 AI 输出短清单，不要写成长文章：

```text
请基于本课程经历，帮我整理 6 份短检查清单：

1. 创建 GitHub Issue 前检查；
2. 需求规格确认检查；
3. 开发前边界检查；
4. 运行和页面验证检查；
5. diff review 检查；
6. 测试失败反馈检查。

要求：

- 每份清单 5 条以内；
- 用我能实际复制使用的语言；
- 不写概念长文；
- 不新增新流程或复杂工具。
```

## 4. 沉淀提示词

让 AI 把真实提示词整理成模板：

```text
请基于本课程经历，帮我整理 5 个可复用提示词模板：

1. 创建 Issue；
2. 收敛需求规格；
3. 执行开发任务；
4. 看 diff；
5. 根据测试或报错反馈修复。

要求：

- 每个模板都要有边界约束；
- 每个模板都要提醒 AI 先复述规格和检查点；
- 每个模板都要适合非开发人员使用；
- 不写成很长的提示词合集。
```

## 5. 形成项目规则

最后沉淀几条最重要的规则：

```text
请基于本课程经历，帮我总结 8 条 AI coding 项目规则。

要求：

- 每条规则都要短；
- 规则要能帮助我控制范围、验证结果、沉淀经验；
- 至少包含 Issue、规格、检查点、diff、测试、范围扩张、复盘；
- 不要写空泛口号。
```

本章要建立的习惯是：每一次 AI coding 结束后，都要把有效经验变成下次可复用的检查清单、提示词或规则。
