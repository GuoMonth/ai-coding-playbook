# 自我验证

## 完成清单

- [ ] 我创建或准备创建了小范围重构评估 Issue；
- [ ] 评估 Issue 写清楚了为什么可能需要重构；
- [ ] 评估 Issue 可以用未来需求说明问题，但没有直接实现新功能；
- [ ] 评估 Issue 写清楚了最小重构目标；
- [ ] 评估 Issue 写清楚了重构前后必须保持不变的行为；
- [ ] 我创建或准备创建了重构开发 Issue；
- [ ] 开发 Issue 引用了评估 Issue；
- [ ] 重构没有新增业务功能；
- [ ] 重构没有改变既有业务规则；
- [ ] 已有测试或人工检查通过；
- [ ] diff 能说明结构更清楚；
- [ ] 本章没有全面重写项目、追求完美架构或引入复杂框架；
- [ ] 本章没有引入后端服务、真实数据库、登录、权限或多租户；
- [ ] 我看过本章 diff，并能说清楚重构改了什么、没改什么。

## 自检问题

如果你能回答下面的问题，就可以进入下一章：

1. 重构和重写有什么区别？
2. 为什么重构前要先说清楚理由？
3. 重构前后哪些行为必须保持不变？
4. 为什么重构时更需要看 diff？
5. 如果 AI 借重构新增业务功能，应该怎么处理？
6. 什么情况下应该先不重构？
7. 这次重构让后续开发哪里更容易？

## 进入下一章前

进入下一章前，你只需要保留一个意识：

```text
重构的价值不在于代码看起来高级，而在于让后续变化更容易，并且不破坏已有行为。
```
