# 学习

本章训练一个常见工程判断：什么时候应该让 AI 重构，什么时候应该先忍住。

## 1. 创建评估 Issue

复制下面的提示词给 AI：

```text
请通过 gh CLI 创建一个 GitHub Issue，用来评估“是否需要做一次小范围重构”。

这个 Issue 只做评估，不直接开发。

请在 Issue 中记录：

- 当前项目里哪里出现了重复、难读或难改；
- 是否有一个未来需求能暴露这个问题；
- 这些问题是否已经影响后续开发；
- 可以选择的最小重构目标；
- 重构后必须保持哪些行为不变；
- 需要运行哪些测试或人工检查；
- 不做全面重写、不追求完美架构、不引入大型状态管理或复杂框架、不新增业务功能、不改变既有业务规则、不引入后端服务、真实数据库、登录、权限或多租户；
- 需要形成最终规格和验证检查点。

创建前请先给我看标题和正文，等我确认后再运行 gh 命令。
```

## 2. 让 AI 先给重构理由

重构前先要求 AI 说明“为什么值得改”：

```text
请基于小范围重构评估 Issue，先不要写代码。

请帮我分析：

1. 当前最明显的重复或职责混乱在哪里；
2. 哪个未来需求能暴露这个问题；
3. 为什么本章不直接实现这个未来需求；
4. 哪个重构目标最小、最值得做；
5. 重构前后哪些业务行为必须保持不变；
6. 应该运行哪些测试或人工检查；
7. 哪些方案属于过度重构；
8. 验证检查点。

请用非开发人员能理解的语言输出。
```

如果 AI 说不出具体收益，就先不要重构。

## 3. 创建开发 Issue 并执行重构

确认后，再创建开发 Issue：

```text
请通过 gh CLI 创建一个 GitHub Issue，用来执行“一次小范围重构”。

要求：

- 引用小范围重构评估 Issue；
- 按已确认范围执行；
- 重构目标必须具体；
- 重构前后业务行为保持不变；
- 运行已有测试或必要人工检查；
- diff 中应能看出结构变清楚，而不是功能变多；
- 不做全面重写、不追求完美架构、不引入大型状态管理或复杂框架、不新增业务功能、不改变既有业务规则、不引入后端服务、真实数据库、登录、权限或多租户。

创建前请先给我看标题和正文，等我确认后再运行 gh 命令。
```

确认后让 AI 执行：

```text
请根据小范围重构开发 Issue 执行实现。

要求：

- 开发前先复述重构目标、保持不变的行为和验证检查点；
- 小步修改；
- 不新增业务功能；
- 不改变既有业务规则；
- 完成后运行测试或必要检查；
- 如果失败，优先回到更小范围；
- 最后说明创建或修改了哪些文件。
```

## 4. 验证重构

重构验证的重点是“行为没变”：

```text
请按小范围重构 Issue 的验证检查点，带我验证：

- 之前能用的商品、库存、订单流程是否仍然能用；
- 已有测试是否通过；
- 页面关键流程是否仍然符合规格；
- diff 中是否没有新增业务功能；
- 结构是否比重构前更清楚；
- 是否没有引入复杂框架或大范围重写。
```

## 5. 看 diff

让 AI 解释重构影响：

```text
请查看本章相关 diff，用非开发人员能理解的方式说明：

- 哪些改动只是移动或抽取逻辑；
- 哪些业务行为保持不变；
- 重构后后续开发会更容易在哪里；
- 有没有风险点；
- 有没有超出本章边界。
```

本章要建立的习惯是：重构是一种有理由、有边界、有验证的结构调整，不是把项目重新写一遍。
