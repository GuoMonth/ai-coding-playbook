# 15. Refactoring

本章目标是通过一个小重构理解技术债和职责边界。

完成本章后，你应该知道：

- 重构不是重写；
- 能跑的代码也可能越来越难改；
- 重构前要先说明为什么需要重构；
- 重构必须保护已有行为；
- diff 和测试是判断重构是否安全的重要反馈。

## 背景

前面章节为了快速学习，很多实现会偏简单。随着商品、库存、订单、退单和测试加入，早期简单写法可能开始变得重复或难以理解。

你也可以用一个未来新增需求作为压力测试，观察当前结构哪里会变难改。但本章不直接实现这个新需求。

本章不追求完美架构，只做一次小范围重构。

## 实现边界

本章只做有明确理由的小范围重构，可以用未来需求辅助判断，但不把未来需求做成新功能。

本章明确不做：

- 全面重写项目；
- 追求完美架构；
- 引入大型状态管理或复杂框架；
- 新增业务功能；
- 改变既有业务规则；
- 后端服务、真实数据库、登录、权限、多租户；
- 为了重构而重构。

## 本章最终结果

完成本章后，项目应该有一次可解释的小重构，例如：

- 抽出订单状态判断；
- 抽出库存计算逻辑；
- 抽出接口契约或 mock 服务的重复代码；
- 整理重复的验证逻辑；
- 保持页面行为不变；
- 保持已有测试或检查通过。

本章重点是重构判断和验证，不是架构设计比赛。
