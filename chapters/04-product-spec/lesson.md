# 学习

本章用商品规格引入一个重要意识：字段不是随便加的，它会影响后面的库存和订单。

## 1. 创建需求 Issue

复制下面的提示词给 AI：

```text
请通过 gh CLI 创建一个 GitHub Issue，用来讨论“商品规格”需求。

这个 Issue 只做需求讨论，不直接开发。

请在 Issue 中记录：

- 为什么商品需要规格；
- 本章最少保留哪些规格字段；
- 哪些字段必填；
- 规格信息应该在哪里展示；
- 当前规格只用于展示和后续任务铺垫；
- 不做完整 SKU、价格、条码、库存流水、订单、数据持久化、API、数据库、登录、权限或多租户；
- 需要形成最终规格和验证检查点。

创建前请先给我看标题和正文，等我确认后再运行 gh 命令。
```

## 2. 讨论字段设计

让 AI 帮你把规格字段讲清楚：

```text
请基于商品规格需求 Issue，先不要写代码。

请用非开发人员能理解的语言解释：

1. 商品和商品规格的区别；
2. 本章为什么只做最小规格；
3. 规格名称、单位、编码或备注分别有什么作用；
4. 哪些字段需要校验；
5. 本章不做完整 SKU 的原因；
6. 验证检查点。

请同时指出这个简化设计未来可能遇到的限制。
```

这里允许 AI 讲一点“以后会复杂”，但不能让它现在就实现复杂模型。

## 3. 创建开发 Issue 并实现

需求确认后，再创建开发 Issue：

```text
请通过 gh CLI 创建一个 GitHub Issue，用来实现“商品规格信息”。

要求：

- 引用商品规格需求 Issue；
- 按已确认规格实现；
- 创建商品时可以填写最小规格信息；
- 商品列表或详情区域能展示规格信息；
- 必填规格字段要有校验；
- 数据可以继续保存在页面内临时状态；
- 不做完整 SKU、价格、条码、入库、订单、数据持久化、API、数据库、登录、权限或多租户。

创建前请先给我看标题和正文，等我确认后再运行 gh 命令。
```

确认后让 AI 执行：

```text
请根据商品规格开发 Issue 执行实现。

要求：

- 开发前先复述最终规格和验证检查点；
- 只改商品规格所需的最小文件；
- 不引入复杂 SKU 或库存订单逻辑；
- 完成后运行项目或必要检查；
- 如果失败，先说明错误类别，再修复；
- 最后说明创建或修改了哪些文件。
```

## 4. 验证规格

重点验证字段和展示：

```text
请按商品规格 Issue 的验证检查点，带我验证：

- 创建商品时是否能填写规格；
- 必填规格为空时是否有提示；
- 保存后是否能看到规格信息；
- 规格是否没有触发库存或订单逻辑；
- 是否出现了完整 SKU、价格或条码等本章不做的内容。
```

## 5. 看 diff

让 AI 解释数据结构变化：

```text
请查看本章相关 diff，用非开发人员能理解的方式说明：

- 商品数据结构发生了什么变化；
- 哪些改动和表单字段有关；
- 哪些改动和展示有关；
- 有没有超出本章边界；
- 这个简化规格设计未来可能在哪些地方不够用。
```

本章要建立的习惯是：新增字段前，先问它属于什么对象、解决什么问题、会影响什么后续流程。
