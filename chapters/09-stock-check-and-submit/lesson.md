# 学习

本章开始处理真正的业务规则：订单能不能提交，要看库存是否足够。

## 1. 创建需求 Issue

复制下面的提示词给 AI：

```text
请通过 gh CLI 创建一个 GitHub Issue，用来讨论“订单提交与库存校验”需求。

这个 Issue 只做需求讨论，不直接开发。

请在 Issue 中记录：

- 提交订单要改变哪些数据；
- 如何判断每个商品库存是否足够；
- 库存不足时如何提示；
- 库存足够时订单状态如何变化；
- 库存足够时库存如何扣减；
- 已提交订单是否可以再次提交；
- 不做退单、完整幂等、并发控制、支付、客户、物流、API、后端数据库、登录、权限或多租户；
- 需要形成最终规格和验证检查点。

创建前请先给我看标题和正文，等我确认后再运行 gh 命令。
```

## 2. 收敛成功和失败路径

让 AI 把提交规则拆成可验证场景：

```text
请基于订单提交与库存校验需求 Issue，先不要写代码。

请帮我收敛：

1. 提交订单的前置条件；
2. 库存不足的失败路径；
3. 库存足够的成功路径；
4. 订单状态如何变化；
5. 商品库存如何变化；
6. 已提交订单再次提交应该如何处理；
7. 本章明确不做的内容；
8. 验证检查点。

请用非开发人员能理解的语言解释“一个动作影响多个数据”。
```

如果 AI 直接进入事务、锁或分布式一致性，让它先保留概念，不在本章实现。

## 3. 创建开发 Issue 并实现

需求确认后，再创建开发 Issue：

```text
请通过 gh CLI 创建一个 GitHub Issue，用来实现“订单提交与库存校验”。

要求：

- 引用订单提交与库存校验需求 Issue；
- 按已确认规格实现；
- 提交前检查订单中每个商品的库存；
- 库存不足时阻止提交并提示具体原因；
- 库存足够时订单状态变为已提交；
- 库存按订单明细数量扣减；
- 已提交订单不应再次作为普通草稿提交；
- 不做退单、完整幂等、并发控制、支付、客户、物流、API、后端数据库、登录、权限或多租户。

创建前请先给我看标题和正文，等我确认后再运行 gh 命令。
```

确认后让 AI 执行：

```text
请根据订单提交与库存校验开发 Issue 执行实现。

要求：

- 开发前先复述最终规格和验证检查点；
- 只改提交订单和库存校验所需的最小文件；
- 不引入退单、并发控制或后端服务；
- 完成后运行项目或必要检查；
- 如果失败，先说明错误类别，再修复；
- 最后说明创建或修改了哪些文件。
```

## 4. 验证提交

本章必须同时验证失败路径和成功路径：

```text
请按订单提交与库存校验 Issue 的验证检查点，带我验证：

- 库存不足时是否阻止提交；
- 库存不足提示是否能说明原因；
- 库存足够时订单是否变为已提交；
- 库存是否按明细数量减少；
- 已提交订单是否不能再次普通提交；
- 是否没有出现退单、支付、客户、物流或后端能力。
```

## 5. 看 diff

让 AI 解释影响范围：

```text
请查看本章相关 diff，用非开发人员能理解的方式说明：

- 哪些改动负责库存检查；
- 哪些改动负责订单状态变化；
- 哪些改动负责库存扣减；
- 成功路径和失败路径分别在哪里体现；
- 为什么本章还不完整解决重复提交和并发问题。
```

本章要建立的习惯是：业务动作要同时看前置条件、状态变化、失败路径和验证方式。
