# 学习

本章只让订单“被看见”。不要急着创建订单，更不要直接做库存扣减。

## 1. 创建需求 Issue

复制下面的提示词给 AI：

```text
请通过 gh CLI 创建一个 GitHub Issue，用来讨论“订单列表页面”需求。

这个 Issue 只做需求讨论，不直接开发。

请在 Issue 中记录：

- 订单列表页面要解决什么问题；
- 订单最少需要展示哪些字段；
- 订单状态有哪些最小枚举；
- 空状态应该如何提示；
- 是否需要简单状态筛选；
- 可以先使用示例订单数据，或只展示空状态；
- 不做创建订单、编辑明细、提交订单、库存扣减、退单、支付、客户、物流、API、后端数据库、登录、权限或多租户；
- 需要形成最终规格和验证检查点。

创建前请先给我看标题和正文，等我确认后再运行 gh 命令。
```

## 2. 收敛订单字段

让 AI 帮你把订单列表讲清楚：

```text
请基于订单列表需求 Issue，先不要写代码。

请帮我收敛：

1. 订单列表最小字段；
2. 订单状态的最小集合；
3. 明细摘要如何展示；
4. 空状态和状态筛选；
5. 本章明确不做的内容；
6. 验证检查点。

请用非开发人员能理解的语言说明“订单记录”和“订单操作”的区别。
```

这里要让 AI 保持克制：本章只看订单记录，不处理订单动作。

## 3. 创建开发 Issue 并实现

需求确认后，再创建开发 Issue：

```text
请通过 gh CLI 创建一个 GitHub Issue，用来实现“订单列表页面”。

要求：

- 引用订单列表需求 Issue；
- 按已确认规格实现；
- 从页面能进入订单列表；
- 订单列表显示订单编号、状态、明细摘要和创建时间；
- 有空状态；
- 有简单状态筛选；
- 可以使用示例订单数据，或只展示空状态；
- 不做创建订单、编辑明细、提交订单、库存扣减、退单、支付、客户、物流、API、后端数据库、登录、权限或多租户。

创建前请先给我看标题和正文，等我确认后再运行 gh 命令。
```

确认后让 AI 执行：

```text
请根据订单列表开发 Issue 执行实现。

要求：

- 开发前先复述最终规格和验证检查点；
- 只改订单列表所需的最小文件；
- 不引入创建订单或库存变化逻辑；
- 完成后运行项目或必要检查；
- 如果失败，先说明错误类别，再修复；
- 最后说明创建或修改了哪些文件。
```

## 4. 验证列表

订单列表的验证重点是字段、状态和边界：

```text
请按订单列表 Issue 的验证检查点，带我验证：

- 是否能进入订单列表；
- 订单编号、状态、明细摘要、创建时间是否可见；
- 空状态是否可见；
- 状态筛选是否符合规格；
- 是否没有创建订单、提交订单或库存扣减能力。
```

## 5. 看 diff

让 AI 解释订单列表改动：

```text
请查看本章相关 diff，用非开发人员能理解的方式说明：

- 哪些文件和订单列表有关；
- 订单状态在哪里定义或使用；
- 哪些内容是示例数据；
- 有没有超出本章边界；
- 为什么本章只展示订单，不创建订单。
```

本章要建立的习惯是：先把记录看清楚，再设计会改变记录的操作。
