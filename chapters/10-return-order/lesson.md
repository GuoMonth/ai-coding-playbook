# 学习

本章让你第一次看到逆向流程。订单提交扣库存，退单则恢复库存。

## 1. 创建需求 Issue

复制下面的提示词给 AI：

```text
请通过 gh CLI 创建一个 GitHub Issue，用来讨论“退单与库存恢复”需求。

这个 Issue 只做需求讨论，不直接开发。

请在 Issue 中记录：

- 哪些订单可以退回；
- 退回后订单状态如何变化；
- 退回后库存如何恢复；
- 已退回订单再次退回应如何处理；
- 为什么退单不是删除订单；
- 不做部分退货、退款、售后审批、换货、补发、物流、完整幂等、并发控制、API、后端数据库、登录、权限或多租户；
- 需要形成最终规格和验证检查点。

创建前请先给我看标题和正文，等我确认后再运行 gh 命令。
```

## 2. 收敛补偿流程

让 AI 把退单规则拆清楚：

```text
请基于退单与库存恢复需求 Issue，先不要写代码。

请用非开发人员能理解的语言解释：

1. 正向流程和逆向流程的区别；
2. 为什么只有已提交订单可以退回；
3. 退回后订单状态如何变化；
4. 库存如何按订单明细恢复；
5. 为什么退单不能直接删除订单；
6. 本章明确不做的内容；
7. 验证检查点。
```

这里要避免 AI 把本章变成完整售后系统。

## 3. 创建开发 Issue 并实现

需求确认后，再创建开发 Issue：

```text
请通过 gh CLI 创建一个 GitHub Issue，用来实现“退单与库存恢复”。

要求：

- 引用退单与库存恢复需求 Issue；
- 按已确认规格实现；
- 只有已提交订单可以退回；
- 退回后订单状态变为已退回；
- 库存按订单明细数量恢复；
- 已退回订单不能再次普通退回；
- 订单记录不能被删除；
- 不做部分退货、退款、售后审批、换货、补发、物流、完整幂等、并发控制、API、后端数据库、登录、权限或多租户。

创建前请先给我看标题和正文，等我确认后再运行 gh 命令。
```

确认后让 AI 执行：

```text
请根据退单与库存恢复开发 Issue 执行实现。

要求：

- 开发前先复述最终规格和验证检查点；
- 只改退单和库存恢复所需的最小文件；
- 不引入退款、售后审批或物流能力；
- 完成后运行项目或必要检查；
- 如果失败，先说明错误类别，再修复；
- 最后说明创建或修改了哪些文件。
```

## 4. 验证退单

本章要验证退回前后状态：

```text
请按退单与库存恢复 Issue 的验证检查点，带我验证：

- 未提交订单是否不能退回；
- 已提交订单是否可以退回；
- 退回后订单是否变为已退回；
- 库存是否按订单明细数量恢复；
- 已退回订单是否不能再次普通退回；
- 订单记录是否仍然存在；
- 是否没有出现退款、售后审批、物流或客户能力。
```

## 5. 看 diff

让 AI 解释补偿相关改动：

```text
请查看本章相关 diff，用非开发人员能理解的方式说明：

- 哪些改动负责判断订单能否退回；
- 哪些改动负责订单状态变化；
- 哪些改动负责库存恢复；
- 为什么退单不是删除订单；
- 有没有超出本章边界。
```

本章要建立的习惯是：逆向流程也要有规格、前置条件、状态变化和验证检查点。
