# 学习

本章回看一个前面反复排除的内容：登录、权限、多租户。现在可以理解为什么它们重要，也为什么前面一直不做。

## 1. 创建评估 Issue

复制下面的提示词给 AI：

```text
请通过 gh CLI 创建一个 GitHub Issue，用来评估“个人库存订单工具进阶为团队 SaaS 时的身份、权限和租户问题”。

这个 Issue 只做概念评估，不直接开发。

请在 Issue 中记录：

- 当前个人工具默认了哪些简化前提；
- 如果变成团队 SaaS，需要新增哪些概念；
- 用户身份解决什么问题；
- 权限解决什么问题；
- 多租户和数据范围解决什么问题；
- 哪些数据模型可能受影响；
- 不做生产级登录注册、RBAC 权限系统、组织管理、真实多租户隔离、数据迁移、安全合规方案或重写前面所有章节；
- 需要形成一份简短评估和自检问题。

创建前请先给我看标题和正文，等我确认后再运行 gh 命令。
```

## 2. 让 AI 解释复杂度

让 AI 用主线案例来解释：

```text
请基于身份、权限和租户评估 Issue，先不要写代码。

请用非开发人员能理解的语言解释：

1. 当前个人工具为什么可以不登录；
2. 团队 SaaS 为什么必须知道用户是谁；
3. 权限和角色大概解决什么问题；
4. 多租户为什么和数据隔离有关；
5. 如果后补这些能力，哪些已有设计可能要改；
6. 为什么前面章节一直禁止 AI 顺手加入登录和权限；
7. 哪些内容只适合作为后续进阶。
```

这里可以讨论概念，但不要让 AI 修改代码。

## 3. 形成进阶边界文档

如果你希望沉淀一份文档，可以让 AI 创建一个进阶说明 Issue：

```text
请通过 gh CLI 创建一个 GitHub Issue，用来沉淀“身份、权限、多租户进阶边界说明”。

要求：

- 引用前一个评估 Issue；
- 输出一份简短说明；
- 说明当前主线为什么仍然保持个人工具；
- 说明未来进阶时需要重新设计哪些边界；
- 明确本次不写代码；
- 不做生产级登录注册、RBAC 权限系统、组织管理、真实多租户隔离、数据迁移、安全合规方案或重写前面所有章节。

创建前请先给我看标题和正文，等我确认后再运行 gh 命令。
```

然后让 AI 输出：

```text
请根据身份、权限、多租户进阶边界说明 Issue，生成一份简短评估。

要求：

- 只写概念和边界；
- 不给出生产实现方案；
- 不修改代码；
- 用个人工具到团队 SaaS 的变化来解释；
- 最后列出 5 个自检问题。
```

## 4. 进行反向检查

用这一章回头检查前面的边界是否合理：

```text
请帮我回顾前面章节，检查是否有地方过早引入了登录、权限、多租户。

如果有，请说明为什么它会增加学习复杂度。

如果没有，请总结为什么这些能力放到进阶更合适。
```

## 5. 沉淀判断规则

最后让 AI 帮你形成一条规则：

```text
请帮我沉淀一条 AI coding 判断规则：

当 AI 主动提出加入登录、权限、多租户时，我应该如何判断它是必要需求，还是范围扩张？

请给出简短规则和 3 个例子。
```

本章要建立的习惯是：重要能力不一定要马上实现，尤其当它会改变系统边界和数据模型时。
