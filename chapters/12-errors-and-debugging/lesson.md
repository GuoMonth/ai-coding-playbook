# 学习

本章训练 AI coding 中非常关键的能力：把错误变成可用反馈。

## 1. 创建调试 Issue

复制下面的提示词给 AI：

```text
请通过 gh CLI 创建一个 GitHub Issue，用来跟踪“一次小错误的复现与修复”。

这个 Issue 可以选择当前项目中的一个小错误，或先故意制造一个容易复现的小错误。

请在 Issue 中记录：

- 错误现象；
- 复现步骤；
- 期望结果；
- 实际结果；
- 需要收集哪些日志或报错；
- 修复必须保持最小范围；
- 不新增业务功能、不做大范围重构、不做性能优化、不引入监控系统、后端服务、真实数据库、登录、权限或多租户；
- 需要形成验证检查点。

创建前请先给我看标题和正文，等我确认后再运行 gh 命令。
```

## 2. 让 AI 先诊断，不急着改

把现象交给 AI 前，先要求它分类：

```text
请基于这个调试 Issue，先不要改代码。

请帮我：

1. 整理复现步骤；
2. 判断错误可能属于哪类问题；
3. 列出还需要我提供哪些信息；
4. 说明最小修复思路；
5. 说明哪些修复方式可能会扩大范围；
6. 给出修复后的验证检查点。

请用非开发人员能理解的语言输出。
```

如果 AI 一上来就大改结构，先要求它解释为什么不能最小修复。

## 3. 执行最小修复

确认后让 AI 修复：

```text
请根据调试 Issue 执行最小修复。

要求：

- 开发前先复述错误现象、复现步骤和验证检查点；
- 只修改修复这个错误所需的最小文件；
- 不新增业务功能；
- 不做大范围重构；
- 修复后运行项目或必要检查；
- 如果第一次修复失败，先解释失败原因，再继续收敛；
- 最后说明创建或修改了哪些文件。
```

## 4. 验证修复

修复后不要只看“没有报错”，要回到复现步骤：

```text
请按调试 Issue 的验证检查点，带我验证：

- 原来的复现步骤是否已经不能复现错误；
- 期望结果是否出现；
- 相关页面或流程是否仍然正常；
- 是否没有新增业务功能或大范围重构；
- 这次修复有没有可能影响其他章节已有能力。
```

## 5. 沉淀经验

让 AI 帮你把这次错误变成知识：

```text
请基于这次调试过程，帮我沉淀一段复盘：

- 错误属于哪类问题；
- 最初的现象是什么；
- 最有用的反馈信息是什么；
- 最小修复改了什么；
- 下次遇到类似问题，我应该先检查什么；
- 是否需要把它加入项目检查清单。
```

本章要建立的习惯是：不要把报错当成阻碍，要把它当成 AI coding 的反馈材料。
