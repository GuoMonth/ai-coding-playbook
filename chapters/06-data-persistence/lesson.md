# 学习

本章解决一个很现实的问题：刷新页面后，前面创建的数据不应该全部丢失。

## 1. 创建需求 Issue

复制下面的提示词给 AI：

```text
请通过 gh CLI 创建一个 GitHub Issue，用来讨论“本地数据保存”需求。

这个 Issue 只做需求讨论，不直接开发。

请在 Issue 中记录：

- 为什么需要让商品、规格和库存刷新后仍然存在；
- 本章使用最简单的本地保存方式；
- 哪些数据需要保存；
- 页面加载时如何恢复数据；
- 是否需要清空本地数据或重置示例数据；
- 保存失败时如何提示；
- 不做后端服务、真实数据库、账号、权限、多租户、多设备同步、数据导入导出、订单提交、退单或库存扣减；
- 需要形成最终规格和验证检查点。

创建前请先给我看标题和正文，等我确认后再运行 gh 命令。
```

## 2. 理解保存边界

让 AI 帮你区分几个概念：

```text
请基于本地数据保存需求 Issue，先不要写代码。

请用非开发人员能理解的语言解释：

1. 页面状态和持久化数据有什么区别；
2. 本地存储和真实数据库有什么区别；
3. 本章最少需要保存哪些字段；
4. 刷新页面为什么是一个重要验证动作；
5. 清空本地数据入口有什么用；
6. 本章明确不做什么；
7. 验证检查点。
```

这里不用深学数据库，只要先理解“保存下来”是一个独立能力。

## 3. 创建开发 Issue 并实现

需求确认后，再创建开发 Issue：

```text
请通过 gh CLI 创建一个 GitHub Issue，用来实现“本地数据保存”。

要求：

- 引用本地数据保存需求 Issue；
- 按已确认规格实现；
- 保存商品、规格和库存数据；
- 页面刷新后可以恢复这些数据；
- 提供清空本地数据或重置示例数据入口；
- 保存失败时有可理解提示；
- 不做后端服务、真实数据库、账号、权限、多租户、多设备同步、数据导入导出、订单提交、退单或库存扣减。

创建前请先给我看标题和正文，等我确认后再运行 gh 命令。
```

确认后让 AI 执行：

```text
请根据本地数据保存开发 Issue 执行实现。

要求：

- 开发前先复述最终规格和验证检查点；
- 使用教程当前项目中最简单、最少改动的实现方式；
- 不引入后端、数据库、账号或多租户；
- 完成后运行项目或必要检查；
- 必须说明如何通过刷新页面验证；
- 如果失败，先说明错误类别，再修复；
- 最后说明创建或修改了哪些文件。
```

## 4. 验证保存

本章最关键的验证动作是刷新页面：

```text
请按本地数据保存 Issue 的验证检查点，带我验证：

- 创建商品后刷新是否仍然存在；
- 商品规格刷新后是否仍然存在；
- 入库后的库存刷新后是否仍然存在；
- 清空本地数据或重置示例数据是否有效；
- 是否没有引入后端、数据库、账号、权限或多租户。
```

## 5. 看 diff

让 AI 解释保存相关改动：

```text
请查看本章相关 diff，用非开发人员能理解的方式说明：

- 哪些改动负责保存数据；
- 哪些改动负责页面加载时恢复数据；
- 保存了哪些字段；
- 清空或重置数据是怎么触发的；
- 为什么这还不是完整数据库能力。
```

本章要建立的习惯是：只要需求说“下次还能看到”，就已经进入了数据保存问题。
