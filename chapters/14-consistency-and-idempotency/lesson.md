# 学习

本章让你看到一个 AI coding 中很常见的问题：功能单次运行正确，不代表重复运行仍然正确。

## 1. 创建需求 Issue

复制下面的提示词给 AI：

```text
请通过 gh CLI 创建一个 GitHub Issue，用来讨论“重复提交、一致性与幂等”需求。

这个 Issue 只做需求讨论，不直接开发。

请在 Issue 中记录：

- 哪些动作可能被重复触发；
- 重复提交订单可能造成什么问题；
- 重复退单可能造成什么问题；
- 本章如何用最小方式观察和处理重复动作；
- 一致性和幂等分别用一句非技术语言解释；
- 不做分布式事务、消息队列、锁、事务隔离级别、高并发压测、后端架构改造、支付、客户、物流、登录、权限、多租户或完整生产级幂等方案；
- 需要形成最终规格和验证检查点。

创建前请先给我看标题和正文，等我确认后再运行 gh 命令。
```

## 2. 讨论风险场景

让 AI 用业务语言解释概念：

```text
请基于重复提交、一致性与幂等需求 Issue，先不要写代码。

请用非开发人员能理解的语言解释：

1. 什么是重复动作；
2. 重复提交订单为什么危险；
3. 重复退单为什么危险；
4. 一致性在库存和订单里是什么意思；
5. 幂等在这个场景里是什么意思；
6. 本章最小处理方案是什么；
7. 本章明确不做的内容；
8. 验证检查点。
```

这里可以让 AI 提到更复杂的方案，但要明确它们不是本章要实现的内容。

## 3. 创建开发 Issue 并实现

需求确认后，再创建开发 Issue：

```text
请通过 gh CLI 创建一个 GitHub Issue，用来实现“重复提交的最小防护与验证”。

要求：

- 引用重复提交、一致性与幂等需求 Issue；
- 按已确认规格实现；
- 重复提交同一订单不应重复扣减库存；
- 重复退回同一订单不应重复恢复库存；
- 页面应给出明确状态或提示；
- 可以增加少量测试或人工验证步骤；
- 不做分布式事务、消息队列、锁、事务隔离级别、高并发压测、后端架构改造、支付、客户、物流、登录、权限、多租户或完整生产级幂等方案。

创建前请先给我看标题和正文，等我确认后再运行 gh 命令。
```

确认后让 AI 执行：

```text
请根据重复提交的最小防护与验证开发 Issue 执行实现。

要求：

- 开发前先复述最终规格和验证检查点；
- 只做最小防护和验证；
- 不把本章扩展成后端并发或分布式一致性专题；
- 完成后运行项目、测试或必要检查；
- 如果失败，先说明错误类别，再修复；
- 最后说明创建或修改了哪些文件。
```

## 4. 验证重复动作

本章必须故意重复操作：

```text
请按重复提交、一致性与幂等 Issue 的验证检查点，带我验证：

- 重复点击提交订单时库存是否只扣一次；
- 重复点击退单时库存是否只恢复一次；
- 页面状态或提示是否清楚；
- 订单状态和库存数量是否对得上；
- 是否没有引入复杂后端架构、锁、消息队列或多租户能力。
```

## 5. 看 diff

让 AI 解释最小防护：

```text
请查看本章相关 diff，用非开发人员能理解的方式说明：

- 哪些改动防止重复扣库存；
- 哪些改动防止重复恢复库存；
- 哪些检查体现了订单和库存的一致性；
- 这个方案为什么只是入门级；
- 生产环境还可能需要哪些更复杂能力，但本章为什么不做。
```

本章要建立的习惯是：凡是会改变关键数据的动作，都要主动问“如果重复发生会怎样”。
