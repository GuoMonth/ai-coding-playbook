# 学习

本章开始训练“操作会改变状态”的意识。手动入库看起来简单，但它已经涉及数量、校验和结果验证。

## 1. 创建需求 Issue

复制下面的提示词给 AI：

```text
请通过 gh CLI 创建一个 GitHub Issue，用来讨论“手动入库”需求。

这个 Issue 只做需求讨论，不直接开发。

请在 Issue 中记录：

- 手动入库要解决什么问题；
- 入库时需要选择什么商品；
- 入库数量有什么限制；
- 入库成功后库存如何变化；
- 页面如何让用户确认库存已经增加；
- 可以显示最近一次入库记录或提示；
- 数据可以继续保存在页面内临时状态；
- 不做订单扣减、退单恢复、库存预占、完整库存流水、数据持久化、API、数据库、登录、权限、多租户、并发或幂等；
- 需要形成最终规格和验证检查点。

创建前请先给我看标题和正文，等我确认后再运行 gh 命令。
```

## 2. 收敛库存变化规则

让 AI 帮你把入库规则讲清楚：

```text
请基于手动入库需求 Issue，先不要写代码。

请帮我收敛：

1. 入库操作的最小流程；
2. 数量字段的校验规则；
3. 入库前后库存如何展示；
4. 成功和失败提示；
5. 本章明确不做的内容；
6. 验证检查点。

请用非开发人员能理解的语言解释“状态变化”是什么意思。
```

如果 AI 开始讨论订单扣库存、并发或幂等，让它先回到本章最小范围。

## 3. 创建开发 Issue 并实现

需求确认后，再创建开发 Issue：

```text
请通过 gh CLI 创建一个 GitHub Issue，用来实现“手动入库”。

要求：

- 引用手动入库需求 Issue；
- 按已确认规格实现；
- 可以选择一个已有商品；
- 可以输入入库数量；
- 数量必须大于 0；
- 提交后商品库存增加；
- 页面能看到入库结果；
- 数据可以继续保存在页面内临时状态；
- 不做订单扣减、退单恢复、库存预占、完整库存流水、数据持久化、API、数据库、登录、权限、多租户、并发或幂等。

创建前请先给我看标题和正文，等我确认后再运行 gh 命令。
```

确认后让 AI 执行：

```text
请根据手动入库开发 Issue 执行实现。

要求：

- 开发前先复述最终规格和验证检查点；
- 只改手动入库所需的最小文件；
- 不引入订单、退单或持久化能力；
- 完成后运行项目或必要检查；
- 如果失败，先说明错误类别，再修复；
- 最后说明创建或修改了哪些文件。
```

## 4. 验证库存变化

入库必须验证“变化前”和“变化后”：

```text
请按手动入库 Issue 的验证检查点，带我验证：

- 入库前商品库存是多少；
- 输入 0 或负数时是否被阻止；
- 输入合法数量后库存是否增加；
- 页面是否能看到入库成功反馈；
- 是否没有出现订单扣减、退单或数据持久化能力。
```

## 5. 看 diff

让 AI 解释库存变化相关改动：

```text
请查看本章相关 diff，用非开发人员能理解的方式说明：

- 哪些改动和入库表单有关；
- 哪些改动改变了商品库存；
- 数量校验在哪里体现；
- 有没有超出本章边界；
- 为什么本章还不讨论订单扣减和幂等。
```

本章要建立的习惯是：凡是会改变数据的操作，都要有输入规则、状态变化和验证检查点。
