# 学习

本章不急着做后端，而是先学习“契约”。契约清楚，AI 才知道页面和数据服务之间应该如何对接。

## 1. 创建需求 Issue

复制下面的提示词给 AI：

```text
请通过 gh CLI 创建一个 GitHub Issue，用来讨论“API 与接口契约”需求。

这个 Issue 只做需求讨论，不直接开发。

请在 Issue 中记录：

- 为什么页面和数据服务需要分工；
- 本章最少需要定义哪些接口；
- 每个接口的请求字段和响应字段；
- 成功响应和错误响应如何表达；
- 是否使用 mock 或最小服务示例；
- 不做完整后端、真实数据库、登录、权限、多租户、微服务、网关、鉴权体系、性能优化或大规模重写；
- 需要形成最终规格和验证检查点。

创建前请先给我看标题和正文，等我确认后再运行 gh 命令。
```

## 2. 收敛契约

让 AI 先解释契约，而不是直接写实现：

```text
请基于 API 与接口契约需求 Issue，先不要写代码。

请用非开发人员能理解的语言解释：

1. API 是什么；
2. 接口契约是什么；
3. 请求、响应、字段、错误分别是什么意思；
4. mock 服务有什么用；
5. 本章最少应该定义哪些接口；
6. 本章明确不做的内容；
7. 验证检查点。

请输出一份最小契约草案。
```

这里的重点是“约定”，不是追求技术完整性。

## 3. 创建开发 Issue 并实现

需求确认后，再创建开发 Issue：

```text
请通过 gh CLI 创建一个 GitHub Issue，用来实现“最小 API 契约和 mock 数据服务”。

要求：

- 引用 API 与接口契约需求 Issue；
- 按已确认规格实现；
- 定义获取商品列表、创建商品、提交订单、退回订单的最小契约；
- 说明请求字段、响应字段和错误字段；
- 可以用 mock 或最小服务函数模拟接口；
- 只做最小改造，让页面能按契约使用或至少能独立验证契约；
- 不做完整后端、真实数据库、登录、权限、多租户、微服务、网关、鉴权体系、性能优化或大规模重写。

创建前请先给我看标题和正文，等我确认后再运行 gh 命令。
```

确认后让 AI 执行：

```text
请根据最小 API 契约和 mock 数据服务开发 Issue 执行实现。

要求：

- 开发前先复述最终规格和验证检查点；
- 优先做最小改动；
- 不把本章扩展成后端项目；
- 完成后运行项目或必要检查；
- 如果失败，先说明错误类别，再修复；
- 最后说明创建或修改了哪些文件。
```

## 4. 验证契约

本章验证的不只是页面，还包括契约是否清楚：

```text
请按 API 与接口契约 Issue 的验证检查点，带我验证：

- 是否能看到最小接口契约；
- 每个接口是否写清请求字段；
- 每个接口是否写清响应字段；
- 错误响应是否有统一表达；
- mock 或最小服务是否按契约返回数据；
- 是否没有引入完整后端、真实数据库、登录、权限或多租户。
```

## 5. 看 diff

让 AI 解释结构变化：

```text
请查看本章相关 diff，用非开发人员能理解的方式说明：

- 哪些文件定义了接口契约；
- 哪些文件模拟了数据服务；
- 页面和数据处理的关系有没有变清楚；
- 哪些改动不是业务功能，而是结构调整；
- 有没有超出本章边界。
```

本章要建立的习惯是：当功能开始变复杂时，先把“约定”写清楚，再让 AI 继续实现。
