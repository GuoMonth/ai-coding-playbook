# 提示词

本章的提示词不是让 AI 写代码，而是让 AI 帮你建立最小工作方式。

你可以直接复制使用。具体命令、参数和操作细节，让 AI 根据你的当前仓库环境判断。

## 1. 让 AI 解释 Git/GitHub 最小概念

```text
我正在学习 AI coding，但我不是专业开发者。

请用非常简洁的方式解释下面这些概念在 AI coding 流程里的作用：

- repository / 代码仓库
- clone
- branch / 分支
- diff
- commit
- pull
- push
- GitHub Issue
- pull request / PR

要求：

- 不要讲 Git 内部原理；
- 不要展开复杂命令；
- 用“AI coding 修改代码时为什么需要它”来解释；
- 最后给我一个从 Issue 到 PR 的最小流程。
```

## 2. 让 AI 解释 gh CLI

```text
请解释一下 GitHub CLI，也就是 gh CLI，在 AI coding 中能做什么。

重点解释：

- 它和 GitHub 网页有什么关系；
- 它能不能创建 Issue；
- 它能不能查看 PR；
- 它能不能帮助 AI 在命令行里操作 GitHub；
- 我作为非开发人员需要知道到什么程度。

请不要让我背命令，只解释最小必要概念。
```

## 3. 让 AI 检查 gh 是否可用

```text
请帮我检查当前环境是否已经安装并登录 GitHub CLI。

要求：

- 先说明你准备运行什么命令；
- 再检查 `gh --version`；
- 再检查 `gh auth status`；
- 如果没有安装或没有登录，请告诉我下一步该怎么处理；
- 如果已经可用，请不要继续创建 Issue，先停下来等我确认。
```

## 4. 让 AI 用 gh 创建第一个 Issue

```text
请通过 gh CLI 在当前 GitHub 仓库创建一个 Issue。

Issue 目标：

- 记录本教程的第一个学习任务；
- 固定练习项目为“个人库存订单管理工具”；
- 用几句话说明：用户创建商品和规格、手动入库、创建订单、提交订单扣减库存、退单恢复库存；
- 说明本章不写代码，只建立 AI coding 工作方式；
- 提醒后续任务会使用 Issue、Git、规格和验证检查点；
- 保持简洁，不要写成复杂项目管理文档。

创建前请先给我看你准备创建的标题和正文。等我确认后，再运行 gh 命令创建。
```

## 5. 让 AI 解释 Issue 的好坏意识

```text
请用产品经理和 UI 设计师能理解的方式解释：

为什么 GitHub Issue 写得好或不好，会影响 AI coding 的效果？

要求：

- 只讲意识，不要给复杂评分标准；
- 用 2 个简单例子说明“模糊 Issue”和“更清楚的 Issue”的差别；
- 告诉我后续章节应该重点观察什么。
```

## 6. 让 AI 解释规格和验证检查点

```text
请解释为什么 AI coding 任务最好有“最小规格”和“验证检查点”。

要求：

- 不要写测试代码；
- 只解释它们为什么能帮助 AI 自我修正；
- 举一个个人库存订单管理工具里的简单例子；
- 告诉我非开发人员需要掌握到什么程度。
```

## 7. 辩论提示词：Issue 是否值得写

正方：

```text
请你扮演一名 AI coding 教练，主张“即使是非开发人员，也应该用 GitHub Issue 记录 AI coding 任务”。

请说明它如何帮助明确边界、保留上下文、沉淀经验。
```

反方：

```text
请你扮演一名追求效率的 AI coding 使用者，反驳“每个任务都要写 GitHub Issue”。

请指出它可能带来的打断、负担和形式主义风险。
```

综合：

```text
综合正反双方观点，请给出一个折中规则：哪些任务值得创建 Issue，哪些小问题可以直接用 prompt 处理？
```

## 8. 辩论提示词：先写规格还是先做页面

正方：

```text
请你扮演一名 AI coding 教练，主张“让 AI 写代码前应该先写最小规格和验证检查点”。

请说明这样为什么更容易让 AI 根据反馈修正。
```

反方：

```text
请你扮演一名 UI 设计师，主张“早期应该先做出可见页面，再慢慢补规格和测试”。

请说明快速看到效果为什么也有价值。
```

综合：

```text
综合正反双方观点，请给出一个适合本教程前几章的折中做法。
```
