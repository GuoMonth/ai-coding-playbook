# 提示词

本章的提示词用于帮助你完成业务对象映射、Issue 描述和学习边界确认。

你可以直接复制使用，也可以替换成自己的表达。

## 1. 让 AI 理解你的学习目标

```text
我正在学习 AI coding 中需要的软件工程基础概念。

我的目标不是成为专业开发者，也不是系统学习某个编程语言，而是希望作为产品经理 / UI 设计师 / 非开发角色，能够理解 AI coding 过程中的反馈、任务边界、验收标准和工程判断。

请你先不要写代码，只帮我检查我的学习场景是否适合后续练习。
```

## 2. 让 AI 帮你选择业务场景

```text
我想选择一个自己熟悉的业务场景，用来学习 AI coding。

候选业务场景如下：

1. [写你的候选场景一]
2. [写你的候选场景二]
3. [写你的候选场景三]

请你从学习软件工程基础概念的角度，帮我判断哪个场景更适合作为教程练习。判断标准是：

- 业务不能太复杂；
- 能逐步做出可见页面；
- 能自然引出列表、表单、数据、流程、权限、错误、测试、重构等概念；
- 适合拆成多个小任务。

请给出推荐，并说明原因。
```

## 3. 让 AI 帮你做业务对象映射

```text
我准备用下面这个业务场景学习 AI coding：

[描述你的业务场景]

教程默认使用“客户、商品、订单、库存”作为示例对象。请你帮我把它们替换成我这个业务中更合适的对象。

请用表格输出：

- 默认对象
- 我的业务对象
- 为什么这样对应
- 如果暂时没有对应对象，建议先如何处理

注意：请不要帮我扩展成复杂业务系统，只做最小必要映射。
```

## 4. 让 AI 帮你生成最小规格和验证检查点

```text
下面是我的业务场景和业务对象映射。请你帮我写一份最小需求规格和验证检查点。

要求：

- 不要扩展成完整系统；
- 只写后续第一两个章节能用到的最小规格；
- 验证检查点要能让 AI coding 在实现后自我检查；
- 检查点可以包括页面可见结果、用户操作、错误提示、测试或命令；
- 输出要适合放进 GitHub Issue。

业务场景和对象映射：

[粘贴你的业务场景和对象映射]
```

## 5. 让 AI 帮你检查 GitHub Issue

```text
下面是我准备创建的 GitHub Issue 内容。请你帮我检查它是否适合作为一个 AI coding 学习任务。

检查重点：

- 背景是否清楚；
- 任务目标是否具体；
- 范围是否足够小；
- 需求规格是否能约束 AI 不乱扩展；
- 验收标准是否能判断完成；
- 测试或验证检查点是否能给 AI 提供反馈；
- 是否混入了暂时不该做的复杂需求；
- 是否适合后续交给 AI coding 工具执行。

请你只做评审和修改建议，不要开始写代码。

Issue 内容：

[粘贴你的 Issue 内容]
```

## 6. 让 AI 帮你压缩任务范围

```text
我担心这个任务范围太大。请你帮我把它拆小。

要求：

- 只保留下一步最小可执行任务；
- 暂时移除复杂业务规则；
- 保留可以验证的可见结果；
- 保留最小规格和验证检查点；
- 用 GitHub Issue 的格式输出；
- 明确“本次做什么”和“本次不做什么”。

原始任务：

[粘贴你的任务描述]
```

## 7. 本章辩论提示词

正方：

```text
请你扮演一名软件工程教练，主张“非开发人员在 AI coding 前应该先用 GitHub Issue 写清楚任务边界和验收标准”。

请用产品经理和 UI 设计师能理解的语言，说明为什么这样做有助于减少返工、提升 AI 输出质量、沉淀经验。
```

反方：

```text
请你扮演一名追求效率的 AI coding 使用者，反驳“每个任务都要先写 GitHub Issue”。

请指出它可能带来的负担、打断和形式主义风险。
```

交换角色后，再问 AI：

```text
综合正反双方观点，请给出一个折中规则：什么时候应该写 GitHub Issue，什么时候可以只用简单 prompt 直接开始？
```

## 8. 规格和测试驱动的辩论提示词

正方：

```text
请你扮演一名 AI coding 教练，主张“在让 AI 写代码前，应该先写清楚需求规格和验证检查点”。

请说明为什么测试、截图、预期输出或运行命令能让 AI 更快发现问题并自我修正。
```

反方：

```text
请你扮演一名产品经理，反驳“每个 AI coding 任务都要先写规格和测试”。

请指出这可能带来的学习负担、前期成本和过度工程化风险。
```

交换角色后，再问 AI：

```text
综合正反双方观点，请给出一个适合非开发人员的折中规则：什么任务必须先写规格和验证检查点，什么任务可以先做可视化原型再补验证？
```
