# 任务：建立最小 AI coding 工作方式

本章不会开发功能。你要做的是让 AI 帮你建立一个最小工作入口：理解 Git/GitHub 的基本位置，并创建第一个 GitHub Issue。

## 任务目标

完成以下三件事：

1. 让 AI 解释本教程会用到的 Git/GitHub 最小概念；
2. 让 AI 解释 `gh` CLI 是什么，以及如何用它创建 Issue；
3. 让 AI 通过 `gh` CLI 创建第一个 Issue。

## 固定练习项目

本教程固定做：

```text
个人库存订单管理工具
```

你暂时不需要设计自己的业务，也不需要扩展更多行业场景。后续章节都会围绕这个固定项目展开。

## 建议创建的第一个 Issue

让 AI 通过 `gh` CLI 创建一个 Issue，标题可以类似：

```text
建立个人库存订单管理工具的 AI coding 学习入口
```

这个 Issue 不需要很完美。它只需要让你第一次看到：一个任务可以被记录在 GitHub Issue 里，并且 Issue 里可以包含目标、范围、规格和验证检查点。

Issue 大致应该说明：

- 本教程要固定做个人库存订单管理工具；
- 这个工具用于创建商品和规格、手动入库、创建订单、提交订单扣减库存、退单恢复库存；
- 第一步不是写代码，而是建立 AI coding 工作方式；
- 后续会用 Issue 记录任务；
- 后续会用 Git 记录代码变化；
- 后续每个任务都尽量有规格和验证检查点。

## 你要避免什么

- 不要一开始设计完整 SaaS；
- 不要一开始扩展多用户、权限或多租户；
- 不要在本章设计复杂订单和库存规则；
- 不要背 Git 命令；
- 不要把 Issue 写成正式项目管理文档；
- 不要在本章开始写功能代码。

## 完成标准

本章完成时，你应该能说清楚：

- 本教程接下来要做什么固定项目；
- GitHub Issue 在 AI coding 中大概起什么作用；
- `gh` CLI 大概能帮你做什么；
- Git 的 commit、push、pull、branch 大概分别在流程里的什么位置；
- 为什么 AI coding 前最好有规格和验证检查点。
