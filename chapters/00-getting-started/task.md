# 任务：建立你的 AI coding 学习闭环

本章任务是为后续章节建立统一工作方式。你不会真正开发功能，但会完成一次“需求到任务”的准备过程。

## 任务目标

完成以下四件事：

1. 选定一个你熟悉的业务场景；
2. 把默认业务对象替换成你自己的业务对象；
3. 准备一个 GitHub Issue 任务模板；
4. 写出最小需求规格和验证检查点；
5. 用 AI 帮你检查任务描述是否清晰。

## 第一步：选择业务场景

选择一个你熟悉、但不需要很复杂的业务场景。

可以是：

- 一个内部管理工具；
- 一个预约或申请系统；
- 一个内容管理工具；
- 一个客户、会员或商户管理工具；
- 一个库存、排期、额度或资源管理工具。

不要一开始选择过大的目标，例如“完整 ERP”“完整电商平台”“完整医院系统”。本教程需要的是一个足够真实、但可以逐步展开的业务软件。

## 第三步：创建第一个 GitHub Issue

在你的仓库里创建一个 Issue，标题可以是：

```text
建立 AI coding 学习闭环和业务对象映射
```

Issue 内容建议使用这个模板：

```md
## 背景

我希望用这个仓库学习 AI coding 中的软件工程基础概念。

## 我的业务场景

这里写你的业务场景。

## 业务对象映射

- 客户 ->
- 商品 ->
- 订单 ->
- 库存 ->

## 本次任务目标

- [ ] 明确业务场景
- [ ] 完成业务对象映射
- [ ] 确认后续章节使用 GitHub Issue 记录任务
- [ ] 写出最小需求规格
- [ ] 写出最小验证检查点
- [ ] 让 AI 检查这个描述是否足够清晰

## 最小需求规格

这个学习项目第一阶段应该帮助我完成：

- 谁使用：
- 解决什么问题：
- 第一个可见页面应该展示：
- 暂时不做：

## 最小验证检查点

- [ ] 我能打开学习项目或页面
- [ ] 我能看到自己选择的业务场景名称
- [ ] 我能看到核心业务对象的名称
- [ ] 如果 AI coding 失败，我能把错误反馈给 AI 继续修正

## 暂时不做

- 不写代码
- 不设计完整业务
- 不追求真实系统复杂度

## 复盘

这次我学到了什么？
```

GitHub Issue 在这里不是项目管理负担，而是任务边界和学习记录。

## 第四步：让 AI 帮你检查

把你的 Issue 内容复制给 AI，让 AI 帮你检查：

- 业务对象是否过大；
- 任务目标是否清晰；
- 需求规格是否足够具体；
- 验证检查点是否能给 AI 反馈；
- 是否有不必要的复杂度；
- 下一章是否可以从“第一个可见页面”开始。

你可以使用 [prompts.md](prompts.md) 里的提示词。

## 完成标准

本章完成时，你应该能清楚回答：

- 我准备用哪个业务场景学习？
- 默认示例中的对象，在我的业务里分别对应什么？
- 我为什么要用 GitHub Issue 记录任务？
- 我为什么要在 AI coding 前写最小规格和验证检查点？
- 后续每一章大致会按什么流程推进？
