# 学习

本章不会开发功能。你要做的是让 AI 帮你建立一个最小工作入口：理解 Git/GitHub 的基本位置，并创建第一个 GitHub Issue。

## 1. 先知道这些东西存在

本章不做系统概念讲解。你只需要知道它们大概解决什么问题：

- 代码仓库：放项目文件的地方；
- Git：记录文件变化；
- branch：一条修改线；
- diff：这次改了什么；
- commit：保存一次修改记录；
- push：把本地修改上传到 GitHub；
- pull：把 GitHub 上的新修改同步到本地；
- GitHub Issue：记录任务、需求或问题；
- PR：请求把一个分支的修改合并回主线；
- `gh` CLI：让 AI 在命令行里操作 GitHub；
- 规格：说明这次到底要做成什么样；
- 验证检查点：判断任务是否完成的检查方式；
- 测试反馈：让 AI 修改后能看到结果，再继续修正。

你现在不需要熟练掌握这些词。先知道它们会反复出现即可。

## 2. 让 AI 解释 Git/GitHub 最小概念

复制这段提示词给 AI：

```text
我正在学习 AI coding，但我不是专业开发者。

请用非常简洁的方式解释下面这些概念在 AI coding 流程里的作用：

- repository / 代码仓库
- clone
- branch / 分支
- diff
- commit
- pull
- push
- GitHub Issue
- pull request / PR

要求：

- 不要讲 Git 内部原理；
- 不要展开复杂命令；
- 用“AI coding 修改代码时为什么需要它”来解释；
- 最后给我一个从 Issue 到 PR 的最小流程。
```

## 3. 让 AI 解释 gh CLI

```text
请解释一下 GitHub CLI，也就是 gh CLI，在 AI coding 中能做什么。

重点解释：

- 它和 GitHub 网页有什么关系；
- 它能不能创建 Issue；
- 它能不能查看 PR；
- 它能不能帮助 AI 在命令行里操作 GitHub；
- 我作为非开发人员需要知道到什么程度。

请不要让我背命令，只解释最小必要概念。
```

## 4. 让 AI 检查 gh 是否可用

```text
请帮我检查当前环境是否已经安装并登录 GitHub CLI。

要求：

- 先说明你准备运行什么命令；
- 再检查 `gh --version`；
- 再检查 `gh auth status`；
- 如果没有安装或没有登录，请告诉我下一步该怎么处理；
- 如果已经可用，请不要继续创建 Issue，先停下来等我确认。
```

## 5. 让 AI 用 gh 创建第一个 Issue

```text
请通过 gh CLI 在当前 GitHub 仓库创建一个 Issue。

Issue 目标：

- 记录本教程的第一个学习任务；
- 固定练习项目为“个人库存订单管理工具”；
- 用几句话说明：用户创建商品和规格、手动入库、创建订单、提交订单扣减库存、退单恢复库存；
- 说明本章不写代码，只建立 AI coding 工作方式；
- 提醒后续任务会使用 Issue、Git、规格和验证检查点；
- 保持简洁，不要写成复杂项目管理文档。

创建前请先给我看你准备创建的标题和正文。等我确认后，再运行 gh 命令创建。
```

## 6. 建立两个意识

Issue 有好坏之分。你现在不需要定义什么是好 Issue，只要知道：Issue 越清楚，AI 越容易理解要做什么。

AI coding 任务最好有规格和验证检查点。没有检查点，AI 写完代码后很难知道自己是否做对。

可以继续用下面的提示词和 AI 讨论：

```text
请用产品经理和 UI 设计师能理解的方式解释：

为什么 GitHub Issue 写得好或不好，会影响 AI coding 的效果？

要求：

- 只讲意识，不要给复杂评分标准；
- 用 2 个简单例子说明“模糊 Issue”和“更清楚的 Issue”的差别；
- 告诉我后续章节应该重点观察什么。
```

```text
请解释为什么 AI coding 任务最好有“最小规格”和“验证检查点”。

要求：

- 不要写测试代码；
- 只解释它们为什么能帮助 AI 自我修正；
- 举一个个人库存订单管理工具里的简单例子；
- 告诉我非开发人员需要掌握到什么程度。
```

## 7. 可选辩论

如果你想更深入一点，可以让 AI 分别扮演正反方。

```text
请你扮演一名 AI coding 教练，主张“即使是非开发人员，也应该用 GitHub Issue 记录 AI coding 任务”。

请说明它如何帮助明确边界、保留上下文、沉淀经验。
```

```text
请你扮演一名追求效率的 AI coding 使用者，反驳“每个任务都要写 GitHub Issue”。

请指出它可能带来的打断、负担和形式主义风险。
```

```text
综合正反双方观点，请给出一个折中规则：哪些任务值得创建 Issue，哪些小问题可以直接用 prompt 处理？
```
