# 00. Getting Started

本章的目标不是写代码，而是建立最小 AI coding 工作方式。

完成本章后，你应该知道：

- 本教程固定使用“个人库存订单管理工具”作为练习项目；
- 代码会放在 GitHub 仓库里；
- Git 用来记录代码变化；
- GitHub Issue 用来记录任务；
- 可以让 AI 通过 `gh` CLI 创建 Issue；
- 一个任务最好有最小规格和验证检查点；
- 测试和反馈能帮助 AI 更快修正问题。

## 固定业务场景

本教程不再要求你一开始选择自己的业务领域。为了降低门槛，所有章节都先围绕一个固定场景展开：

```text
个人库存订单管理工具
```

这个工具可以用几句话理解：

```text
一个人维护自己的商品和库存。
他可以创建商品规格，手动入库，创建包含多个商品的订单。
订单提交时检查并扣减库存，退单时恢复库存。
```

早期只做非常简单的事情：查看商品、创建商品、定义规格、手动入库。后续再逐步加入订单、库存校验、退单、API、测试、重构和部署。多用户、权限和多租户会放到进阶阶段。

固定场景不是因为库存订单管理本身最重要，而是因为它足够常见，适合承载软件工程里的基础概念。

## 本章为什么重要

AI coding 不是把一句需求丢给 AI，然后等它一次性写完。

更好的方式是：

```text
Issue 记录任务 -> 规格说明目标 -> 验证检查点提供反馈 -> AI coding 实现 -> 运行检查 -> AI 根据反馈修正 -> Git 记录变化
```

你现在不需要完全掌握 Git、Issue、测试或规格。先知道它们存在，知道它们分别解决什么问题，再通过后续章节逐步体会。

## 本章要做什么

本章只做三件事：

1. 让 AI 解释本教程会用到的 Git/GitHub 最小概念；
2. 让 AI 解释如何使用 `gh` CLI 创建 GitHub Issue；
3. 让 AI 通过 `gh` CLI 创建本教程的第一个 Issue。

你不需要手写 Issue 模板，也不需要背 Git 命令。你需要形成一个意识：Issue、规格、验证检查点和 Git 记录都会影响 AI coding 的质量。
