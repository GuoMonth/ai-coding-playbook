# 验证

本章不要求你写代码，因此验证重点不是页面效果，而是学习准备是否完成。

## 完成清单

- [ ] 我已经选择了一个熟悉的业务场景；
- [ ] 我能说清楚这个场景为什么适合逐步练习；
- [ ] 我完成了“客户、商品、订单、库存”的业务对象替换；
- [ ] 我知道哪些对象暂时没有对应关系，可以后续再补；
- [ ] 我创建了第一个 GitHub Issue，或者准备好了可以创建的 Issue 内容；
- [ ] Issue 中包含背景、目标、范围、验收标准和复盘位置；
- [ ] Issue 中包含最小需求规格；
- [ ] Issue 中包含至少 3 个验证检查点；
- [ ] 我用 AI 检查过这个 Issue 是否清晰；
- [ ] 我知道后续章节会反复使用“需求 -> 任务 -> Issue -> 规格 -> 验证检查点 -> AI coding -> 反馈修正 -> 复盘”的闭环。

## 自检问题

如果你能回答下面的问题，本章就算完成：

1. 我准备用哪个业务场景学习？
2. 我的业务里，哪个对象最像“客户”？
3. 我的业务里，哪个对象最像“商品”或“资源”？
4. 我的业务里，哪个流程最像“订单”？
5. 我的业务里，什么东西会被占用、消耗、限制或统计？
6. 我这次创建的 Issue，能否让另一个人理解我要做什么？
7. 我是否写清楚了本次暂时不做什么？
8. 我是否写出了 AI 可以用来检查自己的验证点？

## 常见未完成状态

下面这些情况说明还不适合进入下一章：

- 业务场景太大，例如一上来就要做完整企业系统；
- 只写了想法，没有写任务目标；
- 没有写验收标准；
- 没有写最小需求规格；
- 没有任何可验证检查点；
- Issue 里混入了多个不相关需求；
- AI 直接开始写代码，但你还没有确认任务边界；
- 你无法解释为什么这个业务对象对应默认示例中的某个对象。

## 进入下一章前

进入下一章前，你应该保留一份可以复用的 Issue 模板。后续每个小任务都可以从这个模板开始。
