# 概念

本章不做系统概念讲解。你只需要先知道这些东西存在，以及它们大概解决什么问题。

更深入的理解，会通过后续章节的任务、提示词、错误反馈和复盘慢慢建立。

## 代码仓库

代码仓库是放项目文件的地方。AI coding 修改的不是抽象想法，而是仓库里的具体文件。

## Git

Git 用来记录文件变化。你可以先把它理解成“软件项目的修改记录系统”。

本章只需要知道几个词：

- `branch`：一条修改线；
- `diff`：这次改了什么；
- `commit`：保存一次修改记录；
- `push`：把本地修改上传到 GitHub；
- `pull`：把 GitHub 上的新修改同步到本地。

你不需要在第一章熟练掌握这些命令，但要知道它们会反复出现。

## GitHub

GitHub 是托管代码仓库和协作记录的平台。

本教程会用到：

- Issue：记录任务、需求或问题；
- PR：请求把一个分支的修改合并回主线；
- 仓库页面：查看文件、Issue、PR 和历史记录。

## gh CLI

`gh` 是 GitHub 的命令行工具。

在 AI coding 中，它很有用，因为 AI 可以在命令行里帮你查看仓库、创建 Issue、查看 PR。你不需要先背 `gh` 命令，可以让 AI 解释并执行。

## Issue

Issue 可以先理解成“任务记录”。

一个 Issue 写得越清楚，AI 越容易理解要做什么。但本章不要求你定义什么是好 Issue。你只需要先意识到：Issue 有好坏之分，后续任务会让你逐渐体会差别。

## 规格

规格是“这次到底要做成什么样”的说明。

它不一定很长。早期可以只写：

- 要做什么；
- 暂时不做什么；
- 页面上应该看到什么；
- 什么情况算完成。

## 验证检查点

验证检查点是用来判断任务是否完成的检查方式。

早期可以很简单：

- 页面能打开；
- 能看到标题；
- 能看到商品列表；
- 点击按钮有反应；
- 错误时有提示。

## 测试和反馈

测试可以先理解成“可重复的检查”。

AI coding 的一个重要优势是：AI 可以修改代码后运行检查，看到报错或失败结果，再继续修正。你不需要一开始会写测试，但要知道测试和反馈会让 AI coding 更可靠。

## Harness

Harness 可以先理解成“让 AI 能检查自己工作结果的环境”。

它可能包括测试命令、浏览器检查、截图、示例数据、日志和构建命令。本章只需要知道这个方向：不要只让 AI 盲写，要逐步给 AI 可验证反馈。
