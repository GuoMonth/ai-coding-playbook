# 概念

本章只引入后续所有章节都会反复使用的基础概念。这里不追求完整定义，只给出够用的第一层理解。

## 需求

需求是你希望软件帮助用户完成的事情。

在 AI coding 中，需求不能只停留在“做一个客户管理”这种大词上。它需要逐步变成更具体的描述：

- 谁要使用；
- 要完成什么动作；
- 页面上应该看到什么；
- 什么情况算完成；
- 哪些暂时不做。

## 任务

任务是从需求中拆出来的一小步可执行工作。

一个好的任务应该足够小，小到 AI 可以理解、实现、验证。比如“做客户管理系统”太大，“做一个客户列表页面，展示姓名、电话和状态”就更适合作为任务。

## 验收标准

验收标准是判断任务是否完成的依据。

对于非开发人员来说，验收标准不需要一开始写得很专业，但至少要能回答：

- 页面上能看到什么；
- 用户可以做什么；
- 输入错误时发生什么；
- 什么情况说明没有完成。

没有验收标准，AI 写完之后你只能凭感觉判断“好像能用”。

## 需求规格

需求规格是比普通需求更清楚的一版说明。

它不一定要很长，但应该说明：

- 目标用户是谁；
- 要解决什么问题；
- 输入是什么；
- 输出是什么；
- 正常情况应该怎样；
- 异常情况应该怎样；
- 暂时不处理哪些情况。

在 AI coding 中，规格的作用是减少 AI 自己猜测。规格越清楚，AI 越容易写出符合预期的实现。

## 测试

测试是用来检查软件行为是否符合预期的方法。

早期你可以把测试理解成“可重复的检查”。它可以是：

- 打开页面并确认看到某些内容；
- 点击按钮并确认出现某个结果；
- 输入错误内容并确认看到错误提示；
- 运行一条命令并确认没有失败；
- 用自动化测试检查某个函数或流程。

测试不是开发者的专属工作。非开发人员也可以先写出“我希望如何检查结果”，再让 AI 帮忙把它变成可执行测试。

## Harness

Harness 可以先理解成“让 AI 能自己验证工作的工具和环境”。

它可能包括测试命令、示例数据、截图对比、浏览器自动化、日志、构建命令等。你不需要在第一章掌握这些工具，但要先理解一个方向：AI coding 的关键不是让 AI 盲写代码，而是给 AI 一个能不断获得反馈的环境。

后续章节会从最简单的页面检查开始，逐步进入测试、错误反馈和自动化验证。

## GitHub Issue

GitHub Issue 可以理解为一个任务卡片。

在本教程中，Issue 的作用不是做复杂项目管理，而是记录：

- 这次要做什么；
- 为什么要做；
- 暂时不做什么；
- 如何判断完成；
- AI coding 后发生了什么；
- 最后学到了什么。

它是工程思维训练工具，不是额外负担。

## Prompt

Prompt 是你交给 AI 的任务说明。

好的 prompt 不只是命令 AI “帮我做一下”，而是提供上下文、目标、边界和验证方式。GitHub Issue 可以帮助你准备更稳定的 prompt。

## AI coding

AI coding 指用 AI 辅助完成软件实现、修改、调试和解释。

本教程关注的不是“让 AI 一次写对”，而是让你能理解 AI 的反馈，知道如何继续追问、如何验证结果、如何把问题沉淀成经验。

更理想的 AI coding 方式不是“人写一句需求，AI 写完代码就结束”，而是：

```text
人提供规格和验证方式 -> AI 实现 -> AI 运行检查 -> AI 根据反馈修正 -> 人做最终判断
```

## 验证

验证是确认任务真的完成。

早期章节的验证会偏向可视化：页面是否出现、按钮是否能点、表单是否能提交。后期会逐步加入数据、接口、错误、测试和部署等验证方式。

## 复盘

复盘是把一次任务变成下一次经验。

每章结束后都应该回答：

- 这次遇到了什么问题；
- 这个问题属于哪类工程概念；
- 下次应该如何更清楚地描述；
- 哪个检查清单或提示词可以保留下来。
