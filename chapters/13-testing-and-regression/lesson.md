# 学习

本章开始使用测试作为 AI coding 的反馈回路。测试不是为了显得专业，而是为了让错误更早暴露。

## 1. 创建需求 Issue

复制下面的提示词给 AI：

```text
请通过 gh CLI 创建一个 GitHub Issue，用来讨论“最小测试与回归验证”需求。

这个 Issue 只做需求讨论，不直接开发。

请在 Issue 中记录：

- 为什么需要自动化测试；
- 本章优先测试哪些关键业务规则；
- 测试应该如何运行；
- 测试失败时 AI 应该如何处理；
- 哪些人工验证仍然保留；
- 不做全量覆盖、端到端测试体系、CI/CD、复杂测试平台、新增业务功能、大范围重构、后端服务、真实数据库、登录、权限或多租户；
- 需要形成最终规格和验证检查点。

创建前请先给我看标题和正文，等我确认后再运行 gh 命令。
```

## 2. 选择高价值测试

让 AI 先帮你选测试，不要直接铺很多测试：

```text
请基于最小测试与回归验证需求 Issue，先不要写代码。

请帮我选择 2 到 4 个最值得先写的测试。

要求说明：

1. 每个测试覆盖什么规则；
2. 为什么这个规则值得测试；
3. 测试失败时能给 AI 什么反馈；
4. 哪些场景先继续人工验证；
5. 本章明确不做什么；
6. 验证检查点。

请用非开发人员能理解的语言输出。
```

测试选择应该贴近库存、订单提交、退单和表单校验这些高风险规则。

## 3. 创建开发 Issue 并实现

需求确认后，再创建开发 Issue：

```text
请通过 gh CLI 创建一个 GitHub Issue，用来实现“最小测试与回归验证”。

要求：

- 引用最小测试与回归验证需求 Issue；
- 按已确认规格实现；
- 增加少量可运行测试；
- 测试优先覆盖关键业务规则；
- 提供运行测试的命令；
- 如果测试失败，AI 需要根据失败信息修正；
- 不做全量覆盖、端到端测试体系、CI/CD、复杂测试平台、新增业务功能、大范围重构、后端服务、真实数据库、登录、权限或多租户。

创建前请先给我看标题和正文，等我确认后再运行 gh 命令。
```

确认后让 AI 执行：

```text
请根据最小测试与回归验证开发 Issue 执行实现。

要求：

- 开发前先复述最终规格和验证检查点；
- 先说明选择哪些测试以及原因；
- 只加入本章需要的最小测试配置和测试文件；
- 不新增业务功能；
- 运行测试并展示结果；
- 如果测试失败，先解释失败原因，再修复；
- 最后说明创建或修改了哪些文件。
```

## 4. 验证测试反馈

本章的验证重点是测试能不能运行、能不能反馈：

```text
请按最小测试与回归验证 Issue 的验证检查点，带我验证：

- 测试命令是否能运行；
- 测试是否覆盖了已确认的关键规则；
- 测试失败时错误信息是否能帮助定位；
- 修复后测试是否通过；
- 是否没有新增业务功能或大范围重构；
- 哪些场景仍然需要人工看页面验证。
```

## 5. 看 diff

让 AI 解释测试相关改动：

```text
请查看本章相关 diff，用非开发人员能理解的方式说明：

- 哪些文件是测试配置；
- 哪些文件是测试用例；
- 每个测试覆盖什么业务规则；
- 测试如何帮助防止回归；
- 有没有超出本章边界。
```

本章要建立的习惯是：规格讲清楚后，可以把一部分检查点变成可重复运行的测试。
