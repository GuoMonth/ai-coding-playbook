# 16. Deploy And Operate

本章目标是让软件从本地运行走向可访问，并理解发布后的基本反馈。

完成本章后，你应该知道：

- 本地能跑不等于别人能访问；
- 部署前通常需要构建；
- 发布后要做最小冒烟检查；
- 配置和环境会影响运行结果；
- 运行后的反馈也会进入 AI coding 闭环。

## 背景

前面章节都在本地验证。一个工具如果想给别人试用，就需要发布到可访问的环境。

本章只做最小部署意识和操作，不把课程变成 DevOps 教程。

## 实现边界

本章选择最简单的静态部署方式即可，例如 GitHub Pages 或同类静态托管。

本章明确不做：

- 云服务器架构设计；
- Kubernetes、容器编排、灰度发布；
- 监控告警体系；
- 成本规划；
- 后端生产环境；
- 登录、权限、多租户；
- 自动化发布流水线深度建设。

## 本章最终结果

完成本章后，学习者应该能完成一次最小发布：

- 本地构建通过；
- 有明确部署目标；
- 页面可以通过公开或测试链接访问；
- 发布后完成冒烟检查；
- 记录部署失败或页面异常的反馈；
- 知道如何把反馈交给 AI 修复。

本章重点是从“本地可运行”进入“发布后可验证”。
