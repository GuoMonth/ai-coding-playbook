# 学习

本章让你第一次把本地项目发布出去。重点不是掌握平台，而是理解发布前后要验证什么。

## 1. 创建部署 Issue

复制下面的提示词给 AI：

```text
请通过 gh CLI 创建一个 GitHub Issue，用来讨论“最小部署与运行验证”需求。

这个 Issue 只做部署方案和检查点讨论，不直接修改代码。

请在 Issue 中记录：

- 当前项目适合哪种最简单的静态部署方式；
- 部署前需要运行什么构建命令；
- 部署后需要检查哪些页面和流程；
- 如果部署失败，需要收集哪些错误信息；
- 发布后如何把反馈交给 AI 修复；
- 不做云服务器架构、Kubernetes、容器编排、灰度发布、监控告警体系、成本规划、后端生产环境、登录、权限、多租户或复杂发布流水线；
- 需要形成最终规格和验证检查点。

创建前请先给我看标题和正文，等我确认后再运行 gh 命令。
```

## 2. 收敛部署方案

让 AI 帮你选最简单方案：

```text
请基于最小部署与运行验证 Issue，先不要改代码。

请帮我收敛：

1. 当前项目最适合的最小部署方式；
2. 为什么先不要纠结更多平台选择；
3. 部署前构建检查；
4. 部署后冒烟检查；
5. 常见失败反馈应该如何收集；
6. 本章明确不做的内容；
7. 验证检查点。

请用非开发人员能理解的语言输出。
```

这里可以让 AI 推荐最简单路径，但不要把时间花在平台对比上。

## 3. 创建开发 Issue 并完成部署准备

需求确认后，再创建开发 Issue：

```text
请通过 gh CLI 创建一个 GitHub Issue，用来完成“最小部署准备与验证”。

要求：

- 引用最小部署与运行验证需求 Issue；
- 按已确认规格执行；
- 确认本地构建命令可运行；
- 补齐最小部署配置或说明；
- 部署后提供访问链接或明确的验证方式；
- 记录冒烟检查结果；
- 不做云服务器架构、Kubernetes、容器编排、灰度发布、监控告警体系、成本规划、后端生产环境、登录、权限、多租户或复杂发布流水线。

创建前请先给我看标题和正文，等我确认后再运行 gh 命令。
```

确认后让 AI 执行：

```text
请根据最小部署准备与验证 Issue 执行。

要求：

- 执行前先复述部署目标和验证检查点；
- 优先选择最少改动的方式；
- 先运行本地构建；
- 如果需要修改配置，只做部署所需最小改动；
- 部署后做冒烟检查；
- 如果失败，收集错误并分类，再修复；
- 最后说明创建或修改了哪些文件。
```

## 4. 做冒烟检查

发布后检查最核心路径：

```text
请按最小部署与运行验证 Issue 的验证检查点，带我做冒烟检查：

- 首页是否能打开；
- 商品列表是否能打开；
- 创建商品、入库、创建订单、提交订单、退单的核心路径是否至少能访问；
- 页面刷新后是否符合当前数据保存规则；
- 如果有错误，错误信息是什么；
- 是否没有引入复杂云架构、后端生产环境、登录、权限或多租户。
```

## 5. 沉淀运行反馈

让 AI 把发布反馈整理成后续可用材料：

```text
请基于这次部署和冒烟检查，帮我沉淀：

- 部署前检查清单；
- 发布后冒烟检查清单；
- 本次遇到的错误和处理方式；
- 下次发布前应该提前确认什么；
- 哪些内容属于后续进阶，不在本章处理。
```

本章要建立的习惯是：上线不是终点，发布后的反馈也要进入检查、修复和复盘闭环。
