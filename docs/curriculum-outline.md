# 课程大纲

本教程的目标不是让非开发人员学习某个固定业务系统，也不是教授完整的软件开发技能，而是通过一条接近真实软件交付的路径，训练产品经理、UI/UX 设计师等角色在 AI coding 中需要的工程思维。

## 总体定位

教程会围绕一个简化的业务软件展开。默认示例可以是库存、订单和客户管理，但它只是一组教学占位符，不是被固定下来的行业方案。

学习者应该用自己熟悉的业务知识替换这些对象：

- 客户：可以替换成会员、商户、供应商、患者、学生、企业客户；
- 商品：可以替换成服务、课程、项目、资产、内容、设备；
- 订单：可以替换成预约、申请、合同、工单、任务、交易；
- 库存：可以替换成额度、名额、容量、余额、排期、资源占用。

本教程固定的是软件工程过程，开放的是具体业务内容。

## 教学边界

本教程不会替学习者设计完整业务，也不会追求真实 SaaS 的全部复杂性。每个章节只选择足够支撑 AI coding 闭环的概念。

核心边界：

- 不讲大而全的软件工程理论；
- 不以编程语言、框架或工具细节为主线；
- 不要求学习者一次性理解复杂系统；
- 不把默认业务样例变成唯一标准答案；
- 不把 GitHub Issue、分支、测试等工具流程讲成独立工具课。

本教程关注的是：如何把模糊需求变成可执行任务，如何让 AI coding 有清晰上下文，如何验证结果，如何从反馈中沉淀经验。

## 学习闭环

每个具体任务都尽量走同一套闭环：

1. 提出一个业务需求；
2. 用 AI 辅助澄清需求；
3. 把需求拆成可执行任务；
4. 用 GitHub Issue 记录任务边界和验收标准；
5. 写出最小需求规格和可验证检查点；
6. 用提示词让 AI coding 实现任务；
7. 运行测试、检查页面或观察行为变化，让 AI 获得反馈并修正；
8. 根据报错、反馈或 diff 理解相关工程概念；
9. 验证任务是否完成；
10. 复盘并沉淀为下一次可复用的规则。

GitHub Issue 在这里不是项目管理负担，而是工程思维训练工具。它帮助学习者理解任务边界、上下文、验收标准、测试反馈和复盘记录。

## 章节结构

每章建议保持固定文件结构：

- `README.md`：本章目标和完成后的可见结果；
- `task.md`：业务背景、任务说明、范围边界；
- `prompts.md`：可复制给 AI 的提示词；
- `concepts.md`：本章引入的软件工程概念；
- `verify.md`：验证清单和完成标准；
- `reflection.md`：复盘问题、经验沉淀和下一步延伸。

每章只引入少量概念。概念可以先用简化版本帮助理解，后续章节再结合新场景逐步修正和加深。

## 大纲草案

### 00. Getting Started

建立学习方式、AI coding 工作流、GitHub Issue 任务记录方式，以及本教程的业务对象替换方法。

重点概念：需求、任务、需求规格、验收标准、测试反馈、提示词、验证、复盘。

### 01. First Visible Page

做出第一个可见页面，让学习者看到“软件可以被运行和观察”。

默认示例：一个业务软件首页或工作台。

重点概念：项目、页面、组件、运行、预览、可见反馈。

### 02. Core Record List

做一个核心对象列表。

默认示例：客户列表。

可替换对象：会员、供应商、课程、设备、患者、商户等。

重点概念：列表、字段、数据、空状态、加载状态、展示和数据的关系。

### 03. Core Record Form

为核心对象增加新增和编辑能力。

默认示例：新增和编辑客户。

重点概念：表单、用户输入、校验、错误提示、交互状态。

### 04. Managed Resource

引入一个可被管理、占用或消耗的资源。

默认示例：商品和库存。

可替换对象：课程名额、设备库存、服务额度、预约排期、账户余额。

重点概念：业务对象、数量、状态变化、边界条件。

### 05. Business Workflow

把多个对象串成一个业务流程。

默认示例：创建订单并影响库存。

可替换流程：预约、申请、合同审批、工单处理、任务分派。

重点概念：流程、状态流转、前后步骤依赖、完成条件。

### 06. Data Persistence

让数据在刷新或重新打开后仍然存在。

重点概念：本地存储、数据库、数据模型、记录、字段。

### 07. API And Contracts

把页面和数据服务分开，理解前后端如何协作。

重点概念：API、接口契约、请求、响应、字段匹配、错误返回。

### 08. Auth And Scope

引入身份、权限和数据范围。

默认示例：不同团队只能看到自己的数据。

重点概念：登录、权限、角色、租户、数据隔离。

### 09. Errors And Debugging

有意识地面对错误，而不是只让 AI 反复修。

重点概念：报错、日志、复现、定位、最小修改、验证修复。

### 10. Testing And Regression

用测试和检查降低改动风险。

重点概念：验收测试、回归、自动化检查、测试用例、改动影响范围。

### 11. Consistency And Idempotency

引入重复提交、资源扣减、状态不一致等更抽象的问题。

默认示例：重复创建订单或重复扣减库存。

重点概念：一致性、幂等、并发的最小直觉、边界场景。

### 12. Refactoring

通过新增需求暴露早期简单实现的局限，理解为什么需要重构。

重点概念：重构、技术债、重复代码、职责边界、简单实现和长期维护的取舍。

### 13. Deploy And Operate

让软件从本地运行走向可访问。

重点概念：部署、环境变量、配置、发布、运行后反馈。

### 14. Review And Knowledge Base

复盘整个学习过程，把一次次 AI coding 反馈沉淀成个人或团队知识库。

重点概念：代码审查、任务复盘、检查清单、提示词模板、工程判断规则。

## 组织原则

前期优先给视觉反馈，让学习者能看到页面和交互；中期引入数据、流程和接口；后期进入错误处理、测试、幂等、重构和部署等更抽象的工程判断。

每一章都应该允许学习者使用自己的业务对象完成练习。默认示例只负责降低入门门槛，不负责限制想象空间。
