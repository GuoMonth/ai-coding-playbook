# AI Coding Playbook

面向产品经理、UI/UX 设计师、运营、业务分析师等非开发角色的软件工程概念教程。

这个项目的目标不是教人从零开始写代码，也不是整理 AI coding 的提示词技巧，而是帮助已经在软件公司工作、但不具备开发能力的人，在 AI coding 时代建立一套必要的软件工程基础概念。

## 核心目标

AI 降低了写代码和生成软件原型的门槛，但没有自动补齐使用者的软件工程认知。对于非开发角色来说，真正困难的地方往往不是让 AI 写出一段代码，而是在 AI coding 过程中理解反馈、判断问题、验证结果，并把一次次问题沉淀成可复用的经验。

本项目希望提供一套“基座能力”：

- 能听懂 AI coding 过程中常见的软件工程反馈；
- 能判断问题大致属于需求、前端、后端、接口、数据、环境、依赖、构建、测试还是代码结构；
- 能把模糊的产品或设计意图转化为更清晰的工程上下文；
- 能看懂一次代码改动大致影响了什么；
- 能区分“页面看起来能跑”和“实现方式相对可靠、可维护”；
- 能把遇到的问题沉淀为术语、检查清单、排查路径和项目规则。

换句话说，这不是一本编程教程，而是一套帮助非开发人员参与 AI coding、理解 AI coding、积累 AI coding 经验的软件工程通识手册。

## 为什么需要这些概念

AI coding 的反馈通常会涉及大量工程概念，例如：

- 项目结构、依赖、配置、运行环境；
- 前端、后端、API、数据库之间的分工；
- 状态、数据流、缓存、错误态、空状态；
- 构建失败、类型错误、测试失败、接口报错；
- diff、commit、branch、PR、回滚和影响范围；
- 可维护性、重复代码、边界条件和回归风险。

如果缺少这些概念，非开发角色在使用 AI coding 时很容易陷入一种循环：遇到问题就继续把错误粘给 AI，让 AI 修；问题看似解决了，但用户并不知道它属于哪类问题，也不知道下次如何更早识别、更清楚描述、更稳定验证。

本项目关注的正是这个断层：让非开发人员具备足够的概念框架，使他们能够从 AI 的每一次反馈中学习，而不是只把 AI 当作一次性代码生成工具。

## 选取原则

软件工程概念很多，本项目不会追求大而全。进入教程的概念需要满足至少一个条件：

- 能帮助理解 AI coding 的反馈；
- 能帮助定位问题归属；
- 能帮助提出更清晰的 AI coding 请求；
- 能帮助判断 AI 输出的质量；
- 能帮助把一次问题处理沉淀为下一次可复用的经验。

因此，本项目会优先讨论支撑 AI coding 闭环的概念，而不是系统讲授编程语言语法、算法、底层计算机原理或某个框架的完整用法。

## 教学理念

本项目不采用照本宣科的概念讲解方式，而是固定围绕一个极简的“客户与商品管理工具”，从想法、任务、实现、测试到上线迭代，逐步引入必要的软件工程意识。

核心方法包括：

- **场景串联**：用真实软件交付过程组织知识，而不是按概念词典逐条解释；
- **螺旋式学习**：同一个概念会在不同场景中反复出现，先给出够用的简化理解，再逐步补充复杂性；
- **辩论式理解**：通过正反方提示词，让学习者与 AI 围绕工程判断展开讨论，在对比中形成自己的判断规则。

本项目认为，AI 时代的学习重点不是记住更多定义，而是建立可迁移的概念框架和判断能力。教程会尽量避免一次性讲透概念，而是先让学习者意识到 Git、Issue、规格、测试反馈等能力的存在，再通过提示词、AI 对话和自检逐步加深理解。

## 适合谁

这个项目适合：

- 想用 AI coding 做原型、工具或内部系统的产品经理；
- 希望更准确表达交互、状态和边界条件的 UI/UX 设计师；
- 需要与工程团队更高效沟通的软件公司非开发角色；
- 想理解 AI 生成代码质量和风险的人；
- 希望把 AI coding 经验持续沉淀成个人或团队知识库的人。

## 不适合谁

这个项目不试图替代：

- 正统编程语言教程；
- 前端或后端框架教程；
- 计算机科学基础课程；
- 面向专业开发者的工程实践手册。

如果你的目标是成为专业开发者，这些内容可以作为入口，但远远不够。如果你的目标是在 AI coding 时代更好地理解、使用和管理软件实现过程，这个项目正是为你准备的。
